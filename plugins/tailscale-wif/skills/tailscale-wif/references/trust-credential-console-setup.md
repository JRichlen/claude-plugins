# Trust Credential setup — Tailscale admin console walkthrough

The Trust Credential is the object on the Tailscale side that decides whether to
trust a GitHub OIDC token and, if so, what short-lived credential to mint. You
create it once per repo/use-case. It emits a **Client ID** and an **Audience** —
both public — which you then store as repo variables.

Path in the console: **Trust credentials -> Credential -> OpenID Connect**.

## Fields

### Issuer
Choose **"GitHub Actions"** from the preset list. The console fills in the issuer
URL `token.actions.githubusercontent.com` for you. (Selecting the preset also
tells Tailscale how to validate GitHub's signing keys.)

### Subject
This is the GitHub OIDC `sub` claim Tailscale will require. Add a trailing
**`:*`** so a single credential matches **both** event types you care about:

- `...:pull_request` — the test / plan runs on a PR.
- `...:ref:refs/heads/main` — the merge / apply run on `main`.

Two forms exist:

- **Name-only:** `repo:ORG/REPO:*`
- **ID-embedded (rename-proof):** `repo:ORG@<orgId>/REPO@<repoId>:*`

Use the **ID-embedded** form. It is required if your org hardened its OIDC config
(see `oidc-subject-hardening.md`), and it is the better choice regardless because
numeric IDs survive org/repo renames. Compute it with:

~~~sh
skills/tailscale-wif/scripts/compute-subject.sh OWNER/REPO
# or:  gh api repos/OWNER/REPO --jq '{id, owner_id: .owner.id}'
~~~

### Custom claims
Leave **empty** unless you are deliberately narrowing further (e.g. binding to a
specific environment or workflow file). The subject match above is usually
enough.

### Scope
Pick by use case:

- **`policy_file`** — for ACL / policy GitOps (`tailscale/gitops-acl-action`).
  Lets the exchanged token push the tailnet policy.
- **`auth_keys`** — for node join (`tailscale/github-action`). Lets the exchange
  mint an ephemeral auth key so the runner can join the tailnet. With this scope
  you must also assign a **Tag**.

### Tag (auth_keys only)
Assign the tag the runner will bring up as, e.g. **`tag:ci`**. Two things must
both be true or the join fails with `requested tags [tag:ci] are invalid or not
permitted`:

1. The credential is scoped to **assign** that tag (this field).
2. The tag exists in the tailnet policy file's **`tagOwners`**.

### Audience
**Generated by Tailscale** — you do not type it. It has the form
`api.tailscale.com/<client-id>`. Copy it out.

## Output — store as repo variables

The credential gives you a **Client ID** and an **Audience**. Both are public
identifiers, so store them as repo **variables**, not secrets:

```sh
gh variable set TS_OIDC_CLIENT_ID --repo OWNER/REPO --body "<client-id>"
gh variable set TS_OIDC_AUDIENCE  --repo OWNER/REPO --body "api.tailscale.com/<client-id>"
```

Then reference them in the workflow through the `vars` context —
`vars.TS_OIDC_CLIENT_ID` and `vars.TS_OIDC_AUDIENCE` (the templates under
`skills/tailscale-wif/templates/` show the exact YAML). Do not store any
Tailscale API key or OAuth secret alongside them — the whole point of WIF is that
none exists.
